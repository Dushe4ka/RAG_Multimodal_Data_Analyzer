# prod_deep_agent — агент с кратко- и долговременной памятью

CLI-приложение на [DeepAgents](https://github.com/langchain-ai/deepagents) с **кратковременной памятью** (диалог в рамках сессии по `thread_id`) и **долговременной памятью** (файлы в `/memories/*`, хранятся в Postgres между сессиями). Реализация ориентирована на продакшен: все данные в PostgreSQL.

## Запуск

### Требования

- Python 3.10+
- PostgreSQL (для чекпоинтов и store)
- Переменные окружения (см. ниже)

### Установка зависимостей

Из корня репозитория (где лежит общий `myvenv`):

```bash
pip install -r prod_deep_agent/requirements.txt
```

Либо в отдельном venv внутри `prod_deep_agent`:

```bash
cd prod_deep_agent
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### Переменные окружения

Скопируйте пример и задайте свои значения:

```bash
cp prod_deep_agent/.env.example prod_deep_agent/.env
```

Обязательно:

| Переменная        | Описание |
|-------------------|----------|
| `DATABASE_URL`    | Строка подключения к PostgreSQL, например `postgresql://user:pass@localhost:5432/dbname?sslmode=disable` |
| `OPENAI_API_KEY`  | Ключ API для модели (если используете OpenAI-совместимую модель) |

Опционально (есть значения по умолчанию):

| Переменная               | По умолчанию   | Описание |
|--------------------------|----------------|----------|
| `CHAT_MODEL`             | `gpt-4.1-mini` | Модель для диалога |
| `SUMMARY_MODEL`          | как CHAT_MODEL | Модель для суммаризации истории |
| `SUMMARY_TRIGGER_TOKENS` | `4000`         | При каком числе токенов запускать суммаризацию |
| `SUMMARY_KEEP_MESSAGES`  | `20`           | Сколько последних сообщений не суммаризовать |

Файл `.env` ищется в каталоге `prod_deep_agent/` (можно задать переменные и в окружении).

### Запуск CLI

**Из корня репозитория:**

```bash
PYTHONPATH=. python -m prod_deep_agent --help
PYTHONPATH=. python -m prod_deep_agent chat
PYTHONPATH=. python -m prod_deep_agent chat --thread-id my-dialog-1
```

**Из каталога `prod_deep_agent`:**

```bash
cd prod_deep_agent
PYTHONPATH=.. python -m app.cli --help
PYTHONPATH=.. python -m app.cli chat
```

При первом запуске с указанным `DATABASE_URL` выполняются миграции (создание таблиц для чекпоинтов и store).

### Запуск API и веб-интерфейса

Из корня репозитория (агент с MCP-инструментами: context7, duckduckgo и др.):

```bash
PYTHONPATH=. uvicorn prod_deep_agent.api.main:app --reload --host 0.0.0.0 --port 8000
```

Откройте в браузере: **http://localhost:8000/** — веб-чат с возможностями:
- выбор/создание диалога (thread_id);
- отправка сообщений в `/api/chat`;
- история чекпоинтов и откат (`/api/threads/{id}/history`, откат через `from_checkpoint` при следующем сообщении);
- долговременная память: список, чтение, запись (`/api/memory`, `/api/memory/read`, `/api/memory/write`).

Документация API: **http://localhost:8000/docs**.

### Команды

- **`chat`** — интерактивный чат. Кратковременная память привязана к `--thread-id`; долговременная — файлы в `/memories/` (агент может их читать/писать сам).
- **`history --thread-id <id>`** — список чекпоинтов диалога (для отката).
- **`rollback --thread-id <id> --checkpoint-id <id>`** или **`--steps N`** — вывести команду для продолжения чата с выбранного чекпоинта.
- **`memory ls | read <path> | write <path> --content "..."`** — просмотр/запись долговременной памяти (файлы в store). Опционально `--user-id` для неймспейса (память привязана к пользователю).

## Архитектура памяти

- **Кратковременная память**: состояние графа сохраняется через `PostgresSaver`. Ключ — пара **(`thread_id`, `checkpoint_ns`)**, где `checkpoint_ns` задаётся из **`user_id`**. У каждого пользователя свои потоки: у обоих может быть «поток 1», они не пересекаются.
- **Долговременная память**: файловая система агента с роутингом (`CompositeBackend`):
  - пути **без** префикса `/memories/` — эфемерные (в состоянии потока);
  - пути **с** префиксом `/memories/` — персистентные в Postgres через `PostgresStore` (StoreBackend). Namespace в Store — **`user_id`**: у каждого пользователя свой набор данных, несколько потоков одного пользователя видят одну долговременную память.
- **Изоляция по пользователю**: агент при вызове получает только один namespace (по `user_id` из `context`/`config`). Эндпоинты памяти (`/api/memory*`) при наличии заголовка **`X-User-Id`** используют только его — в продакшене заголовок можно выставлять из сессии/авторизации, чтобы клиент не мог запрашивать память другого пользователя.

Откат: выбор прошлого чекпоинта по `history` и продолжение чата с `chat --from-checkpoint <checkpoint_id>`.

## Варианты использования памяти при написании агентов

Краткий справочник паттернов (LangChain / LangGraph / DeepAgents).

### Кратковременная память (один диалог / поток)

- **Обрезка сообщений** — оставлять последние K сообщений или последние X токенов перед вызовом LLM, чтобы не выходить за контекстное окно.
- **Удаление сообщений** — точечно удалять из состояния (`RemoveMessage` по id или `REMOVE_ALL_MESSAGES`) с учётом требований провайдера (например, чередование human/assistant/tool).
- **Суммаризация** — заменять старую часть истории кратким изложением (middleware или отдельный узел), сохраняя последние N сообщений в полном виде.
- **Фильтрация после модели** — в `after_model` удалять или править ответы (например, по политике безопасности).

### Долговременная память (между сессиями)

- **Профиль пользователя в Store** — хранить в store по неймспейсу (user_id, …) предпочтения, имя, контекст; читать в инструментах или в системном промпте.
- **Семантический поиск по Store** — векторный индекс в store; по последнему запросу пользователя подтягивать релевантные факты и подмешивать в контекст.
- **Файловая долговременная память DeepAgents** (`/memories/*`):
  - `/memories/user_preferences.txt` — предпочтения пользователя;
  - `/memories/instructions.txt` — самообучающиеся инструкции по отзывам;
  - `/memories/project/notes.md` — база знаний по проекту;
  - `/memories/research/...` — долгие исследования между сессиями.

### Гибридные стратегии

- Важные факты писать в store (долговременно), остальное сжимать суммаризацией или обрезкой.
- Периодическая очистка устаревших файлов в store для экономии места.

В этом проекте реализованы: кратковременная память (checkpointer + суммаризация), долговременная файловая память `/memories/*` в Postgres и базовый откат по чекпоинтам.
